
## 1. 동시성 문제란?

### 1-1. 멀티 스레드

- 단일 스레드에서는 문제 없음
- 멀티 스레드에서 발생.
- 요청을 동시에 처리하면서,
  Tomcat 기본 스레드 200개, CPU 코어 여러 개가 진짜로 **동시에 실행**된다.
```
스레드 1: 읽기(100) ------------------- 저장(150)
스레드 2:           읽기(100) → 저장(120)

최종 값: 120 ← 스레드 1이 저장한 150이 날아감
```
두 스레드가 같은 데이터를 **동시에 읽고 쓰면** 한쪽 작업이 사라진다.

### 1-2. 동시성 문제가 발생하는 3가지 조건 (Bernstein 조건)

```
1. 공유 자원 존재        → 경매 최고가 (Redis 키 또는 DB 레코드)
2. 최소 하나의 씌기 연산  → 입찰 처리 (최고가 업데이트)
3. 순서 보장 없음        → 여러 스레드가 락 없이 접근
```

세 가지 모두 동시에 성립할 때 동시성 문제 발생함.
[해결책]
- 공유 자원을 없앤다 (불변 객체 / 복사본 사용)
- 쓰기를 원자적으로 만든다 (Atomic 연산)
- 순서를 강제한다 (Lock)

## 2. Race Condition

### 2-1. 정의

**Race Condition**이란, 둘 이상의 실행 흐름이 공유 자원에 동시 접근할 때,
실행 순서(타이밍)에 따라 결과가 달라지는 상황이다.

"경주(Race)"라는 이름처럼, 어떤 스레드가 먼저 도착하냐에 따라 결과가 바뀜

### 2-2. BidFlow에서의 Race Condition

Lock 없이 입찰 처리할 때:
```
초기 상태: 최고가 = 100,000원

스레드 A (유저 1, 입찰가 150,000원)
스레드 B (유저 2, 입찰가 120,000원)

T=1ms  A: Redis에서 최고가 읽기 → 100,000
T=2ms  B: Redis에서 최고가 읽기 → 100,000
T=3ms  A: 150,000 > 100,000 → 검증 통과
T=4ms  B: 120,000 > 100,000 → 검증 통과  ← A가 아직 안 썼으므로 통과
T=5ms  A: Redis에 최고가 = 150,000 저장
T=6ms  B: Redis에 최고가 = 120,000 저장  ← A의 결과를 덮어씀

최종 최고가: 120,000원  ← 150,000원이 정상
```

유저 1이 더 높은 금액을 입찰했지만, 유저 2의 입찰가가 최고가가 되었음.

### 2-3. Race Condition이 항상 재현되지 않는 이유

운 좋게 순서가 맞으면 정상동작함.
개발 환경에서 테스트하면 재현 안 되다가 트래픽 몰리는 실서비스에서 터지는 경우가 있음. 

### 2-4. 재현 코드 (테스트용)

```java
@Test
void raceConditionTest() throws InterruptedException {
	int threadCount = 50;
		CountDownLatch startLatch = new CountDownLatch(1); // 동시 출발
		CountDownLatch doneLatch = new CountDownLatch(threadCount);
		AtomicInteger successCount = new AtomicInteger(0);
		
		// 초기 최고가 설정
		stringRedisTemplate.opsForValue().set("auction:room:1:highestBid", "100000");
		
		for (int i=0; i<threadCount; i++) {
			final long bidAmount = 100000L + (i + 1) * 1000L; // 101,000 ~ 150,000
			executorService.submit(() -> {
				try {
					startLatch.await(); // 모든 스레드가 준비될 때까지 대기
					
					// Lock 없는 경우
					String current = stringRedisTemplate.opsForValue().get("auction:room:1:highestBid");
					long currentBid = Long.parseLong(current);
					
					if (bidAmount > currentBid) {
						// 이 사이에 다른 스레드 끼어들 수 있음
						stringRedisTemplate.opsForValue().set("auction:room:1:highestBid", String.valueOf(bidAmount));
						successCount.incrementAndGet();
					}
				} catch (InterruptedException e) {
					Thread.currentThread().interrupt();
				} finally {
					doneLatch.countDown();
				}
			});
		}
		
		startLatch.countDown();  // 동시 출발
		doneLatch.await();
		
		String finalBid = stringRedisTemplate.opsForValue().get("auction:room:1:highestBid");
		
		System.out.println("성공 횟수: " + successCount.get());
		System.out.println("최종 최고가: " + finalBid);
		// 기대값: 최종 최고가 = 150,000
		// 실제값 : 실행때 마다 다름 → Race Condition 확인
}
```

## 3. Lost Update

### 3-1. Lost Update란?

**Lost Update** 란, 한 트랜잭션의 업데이트가 다른 트랜잭션에 의해 덮어씌워져 사라지는 현상이다.
Race Condition의 결과 중 가장 흔하고 직접적인 데이터 손상 형태

### 3-2. Read-Modify-Write 패턴이 취약한 이유

Lost Update는 항상 이 3단계 패턴에서 발생한다.

```
1. Read : 현재 값을 읽는다.
2. Modify : 애플리케이션에서 새 값을 계산한다.
3. Write : 새 값을 저장한다.
```
-> 여기서, 1~3 사이에 다른 스레드가 끼어들 수 있는 **시간 틈(time window)** 이 존재

### 3-2. Lost Update vs Race Condition 관계

```
Race Condition (넓은 개념)
  ㄴ  Lost Update  (Race Condition의 결과 중 하나)
  ㄴ  Dirty Read   (Race Condition의 다른 형태)
  ㄴ  Phantom Read (Race Condition의 다른 형태)
```

## 4. Dirty Read / Non-Repeatable Read / Phantom Read

DB 트랜잭션에서 정의하는 동시성 이상 현상들.
Redis와 직접 연관은 없으나, DB Lock을 비교할 때 기반이 됨

### 4-1. Dirty Read

**커밋되지 않은 데이터를 읽는 것**. 트랜잭션 격리 수준 `READ UNCOMMITTED`에서 발생

```
T=1  A: 최고가 100,000 → 150,000으로 UPDATE (미커밋)
T=2  B: 최고가 읽기 → 150,000 (A의 미커밋 값)
T=3  A: 롤백 → 최고가 다시 100,000으로 변경
T=4  B: 150,000 기준으로 로직 진행 → 잘못된 결과
```

### 4-2. Non-Repeatable Read

**같은 쿼리를 두 번 실행했을 때 결과가 다른 것**. `READ COMMITTED`에서 발생

```
T=1  A: 최고가 읽기 → 100,000
T=2  B: 최고가 150,000 으로 UPDATE + 커밋
T=3  A: 최고가 다시 읽기 → 150,000  ← 같은 트랜잭션인데 값이 바뀜
```

### 4-3. Phantom Read

**같은 조건의 조회를 두 번 실행했을 때 행(row)의 수가 다른 것**.
`REPEATABLE READ` 에서 발생.

```
T=1  A: "입찰금액 > 100,000" 인 입찰 조회 → 5건
T=2  B: 입찰금액 120,000짜리 새 입찰 INSERT + 커밋
T=3  A: 같은 조건 재조회 → 6건  ← 유령 행(Phantom)이 생김
```

### 4-4. MySQL InnoDB의 기본 격리 수준

MySQL InnoDB는 기본으로 `REPEATABLE READ`를 사용하며,
MVCC(Multi-Version Concurrency Control)로 Phantom Read까지 방어한다.

```
READ UNCOMMITTED  → Dirty Read 허용
READ COMMITTED    → Dirty Read 방지
REPEATABLe READ   → Non-Repeatable Read 방지 (MySQL 기본값)
SERIALIZABLE      → Phantom Read 방지 (가장 강력, 가장 느림)
```

## 5. BidFlow 프로젝트 > 동시성 문제 발생 시나리오

### 시나리오 A: 동시 입찰- Redis, Lock 없음

```
상태: 최고가 100,000원, 경매방 1번

유저 1 (입찰가 150,000): 스레드 1
유저 2 (입찰가 120,000): 스레드 2

1. 스레드 1: Redis GET "auction:room:1:highestBid" → "100000"
2. 스레드 2: Redis GET "auction:room:1:highestBid" → "100000"
3. 스레드 1: 150,000 > 100,000 → 검증 통과
4. 스레드 2: 120,000 > 100,000 → 검증 통과  ← 이미 잘못됨
5. 스레드 2: Redis SET → "120000"
6. 스레드 1: Redis SET → "150000"

최종: 150,000원 (우연히 맞음)

하지만 순서가 바뀌면:
5. 스레드 1: Redis SET → "150000"
6. 스레드 2: Redis SET → "120000"  ← Lost Update!

최종: 120,000
```

→ Race Condition.

### 시나리오 B: DB SELECT FOR UPDATE로만 처리할 때

```java
@Transactional
public void placeBid(Long roomId, Long amount) {
	// Row Lock 획득 (다른 트랜잭션 블로킹)
	AuctionRoom room = auctionRoomRepository.findByIdWithLock(roomId);
	
	if (amount <= room.getHighestBid()) {
		throw new BidAmountInvalidException();
	}
	
	room.updateHighestBid(amount);  // Dirty Checking으로 update
	bidRepository.save(new Bid(...));
}
```

이 방식은 **정합성은 보장**되지만:
```
동시 요청 100개:
	요청 1이 FOR UPDATE 획득 → 나머지 99개 대기
	요청 1 완료 → 요청 2 획득 → 나머지 98개 대기
	...
	
DB 커넥션 10개 x 직렬 처리 → 처리량(QPS) 대폭 감소
트래픽 급증 시 커넥션 풀 고갈 → 타임아웃 → 장애
```

### 시나리오 C: 서버 2대 + Java synchronized

```java
// 단일 서버에서는 동작
public synchronized void placeBid(Long roomId, Long amount) {
	// ...
}
```

`synchronized`는 **같은 JVM 프로세스 안에서만** 동기화 됨.
서버 2대 이상일 시 무의미 → **분산 환경에서는 분산 락 필요**

## 6. Java 레벨에서 해결할 수 있는 방안

**단일 서버 환경**에서의 Java 동시성 도구들.

### 6-1. synchronized

메서드 또는 블록을 **모니터 락**으로 보호한다. 한 번에 하나의 스레드만 진입 가능

```java
// 메서드 전체 보호
public synchronized void placeBid(Long roomId, Long amount) {
	// 한 번에 하나의 스레드만 실행
}

// 블록 단위 보호 (더 세밀한 제어)
public void placeBid(Long roomId, Long amount) {
	// 락 불필요한 작업
	
	synchronized (this) {
		// 공유 자원 접근
	}
	
	// 락 불필요한 작업 ...
}

// 특정 객체를 락 대상으로 지정
private final Object bidLock = new Object();
public void placeBid(Long roomId, Long amount) {
	synchronized (bidLock) {
		// ...
	}
}
```

### 6-2. volatile

변수의 값을 **CPU 캐시가 아닌 메인 메모리에서 직접 읽고 쓰도록** 강제

```
CPU 1 (스레드 A): [레지스터/캐시: highest=100] ----- 메인 메모리: 100
CPU 2 (스레드 B): [레지스터/캐시: highest=100]  ↑
                                                 ↑ volatile이 없으면
                                                 스레드 A가 150으로 바꿔도
                                                 B는 캐시에서 100을 읽음
```

```java
private volatile long highestBid = 0;

// 이제 모든 스레드가 메인 메모리에서 직접 읽고 씀
```

🚨 `volatile`은 가시성(visibility)만 보장한다.
	읽기/쓰기 각각은 원자적이지만 **Read-Modify-Write는 원자적이지 않다.**

```java
// 이건 여전히 Race Condition
if (newBid > highestBid) {    // 읽기 (volatile → 메인 메모리)
	highestBid = newBid;      // 쓰기 (volatile → 메인 메모리)
	// 두 연산 사이에 다른 스레드가 끼어들 수 있음
}
```


### 6-3. AtomicLong / AtomicReference

CAS(Compare-And-Swap) 명령어를 사용해 **원자적 Read-Modify-Write**를 구현

```java
private final AtomicLong highestBid = new AtomicLong(0);

public boolean tryUpdateHighestBid(long newBid) {
	while (true) {
		long current = highestBid.get();    // 현재 값 읽기
		
		if (newBid <= current) {
			return false; // 입찰 실패
		}
		
		// CAS : "현재 값이 current이면 newBid로 교체"
		// 다른 스레드가 current를 바꿨으면 false → while 재시도
		if (highestBid.compareAndSet(current, newBid)) {
			return true; // 원자적 교체 성공
		}
		// CAS 실패 → 루프 처음부터 재시도 (Spin Lock / Busy Waiting)
	}
}
```

### 6-4. CAS 동작 원리

```
CPU 명령어 수준에서 이루어짐 (LOCK CMPXCHG)

compareAndSet(expected=100, update=150):
	1. 메모리 값 읽기 → 100
	2. expected(100)와 비교
	3. 같으면: 메모리를 150으로 교체 + true 반환 (원자적)
	4. 다르면: 아무것도 안 함 + false 반환
	   
이 비교-교체 전체가 CPU가 보장하는 원자 연산
```

```java
// AtomicLong 주요 메서드
AtomicLong counter = new AtomicLong(100);

counter.get();                    // 현재 값 읽기: 100
counter.set(200);                 // 값 설정 (원자적)
counter.getAndIncrement();        // 읽고 1 증가 (이전 값 반환)
counter.incrementAndGet();        // 1 증가하고 읽기 (새 값 반환)
counter.compareAndSet(100, 150);  // CAS
counter.getAndUpdate(v -> v > 100 ? v + 10 : v);   // 조건부 업데이트
```

**한계**: JVM 내부의 원자성만 보장.

### 세 가지 비교

|        | synchronized | volatile      | AtomicLong |
| ------ | ------------ | ------------- | ---------- |
| 가시성 보장 | O            | O             | O          |
| 원자성 보장 | O (블록 전체)    | X (단일 읽기/쓰기만) | O (CAS 연산) |
| 성능     | 낮음 (블로킹)     | 높음            | 높음 (논블로킹)  |
| 사용 케이스 | 복잡한 임계구역     | 단순 플래그        | 숫자 카운터     |
| 분산 환경  | X            | X             | X          |

## 7. DB 레벨 해결법 - 낙관적 락 / 비관적 락

### 7-1. 낙관적 락 (Optimistic Lock)

"충돌이 드물 것" 이라는 가정 하에 **락 없이 작업 후, 커밋 시점에 충돌 감지**

Jpa의 `@Version` 어노테이션으로 구현

```java
@Entity
public class AuctionRoom {
	@Id
	private Long id;
	
	private Long highestBid;
	
	@Version                  // 충돌 감지용 버전 컬
	private Long version;   
}
```

```sql
-- 조회 시 version 포함
SELECT id, highest_bid, version FROM auction_rooms WHERE id = 1;
-- 결과: id=1, highest_bid=100000, version=5

-- 업데이트 시 version 조건 포함
UPDATE auction_rooms
SET highest_bid = 150000, version = 6
WHERE id = 1 AND version = 5;   -- version이 맞아야 업데이트함
-- 다른 스레드가 먼저 version을 6으로 바꿨다면 → 0 rows affected
```

```java
// JPA가 자동 처리 - version 불일치 시 OptimisticLockException 발생
@Transactional
public void placeBid(Long roomId, Long amount) {
	AuctionRoom room = auctionRoomRepository.findById(roomId).orElseThrow();
	
	if (amount <= room.getHighestBid()) {
		throw new BidAmountInvalidException();
	}
	
	room.updateHighestBid(amount); // version 자동 증가
	// 커밋 시점에 다른 스레드가 version을 바꿨으면 → OptimisticLockException
}

// 재시도 로직
public void placeBidWithRetry(Long roomId, Long amount) {
	int maxRetries = 3;
	for (int i=0; i<maxRetries; i++) {
		try {
			placeBid(roomId, amount);
			return;
		} catch (OptimisticLockException e) {
			if (i == maxRetries - 1) throw new BidFailedException("입찰 경합이 심합니다.");
			// 잠시 후 재시도 (지수 백오프 권장)
			Thread.sleep(50 * (long) Math.pow(2, i));
		}
	}
}
```
**적합한 상황**: 충돌이 드문 경우 (읽기 多, 쓰기 少)
**부적합한 상황**: 경매 마감 직전처럼 충돌이 잦은 경우 → 재시도 많아짐

### 7-2. 비관적 락 (Pessimistic Lock)

"충돌이 자주 있을 것"이라는 가정 하에 **처음부터 락을 잡고 작업**한다.

```java
// JPA Repository
@Lock(LockModeType.PERSSIMISTIC_WRITE)
@Query("SELECT r FROM AuctionRoom r WHERE r.id = :roomId")
Optional<AuctionRoom> findByIdWithLock(@Param("roomId") Long roomId);
```

```sql
-- 실행되는 SQL
SELECT * FROM auction_rooms WHERE id = 1 FOR UPDATE;
-- 이 트랜잭션이 커밋/롤백될 때까지 다른 트랜잭션은 같은 행에 FOR UPDATE 불가
```

```java
@Transactional
public void placeBid(Long roomId, Long amount) {
	AuctionRoom room = auctionRoomRepository
		.findByIdWithLock(roomId)     // SELECT FOR UPDATE - 락 획득
		.orElseThrow();
		
	if (amount <= room.getHighestBid()) {
		throw new BidAmountInvalidException();
	}
	
	room.updateHighestBid(amount);
	bidRepository.save(new Bid(...));
	// 트랜잭션 종료 → 락 해제
}
```

**적합한 상황**: 충돌이 잦은 경우, 데이터 정합성이 최우선
**단점**:
- DB 커넥션을 락 보유 동안 점유 → 커넥션 풀 고갈
- Deadlock 가능성 (두 트랜잭션이 서로의 락을 기다림)
- 단일 DB에만 유효 → 샤딩 / 다중 DB 환경에서 무력


### 낙관적 락 vs 비관적 락 비교

| 항목 | 낙관적 락 | 비관적 락 |
|---|---|---|
| 락 획득 시점 | 커밋 시 | 조회 시 |
| 충돌 처리 | 예외 → 재시도 | 다른 트랜잭션 블로킹 |
| 성능 (충돌 적을 때) | 높음 (락 오버헤드 없음) | 낮음 |
| 성능 (충돌 많을 때) | 낮음 (재시도 폭발) | 보통 (직렬 처리) |
| Deadlock 가능성 | 없음 | 있음 |
| DB 커넥션 점유 | 짧음 | 락 보유 동안 점유 |
| 분산 환경 | X (단일 DB) | X (단일 DB) |

## 8. Redis 레벨 해결책 - 원자 커맨드 / 분산 락

### 8-1. Redis 원자 커맨드

충돌 없이 숫자를 증가시키는 경우면 `INCR`/`INCRBY`가 충분
```java
// 입찰 횟수 카운터 - Race Condition 없이
Long totalBids = stringRedisTemplate.opsForValue()
	.increment("auction:room:" + roomId + ":bidCount");
```
단, **조건부 업데이트**에는 사용 불가

### 8-2. Lua 스크립트 - 조건부 원자 연산

Redis는 Lua 스크립트를 **단일 원자 연산**으로 실행함
스크립트 실행 중엔 다른 커맨드가 끼어들 수 없음.

```lua
-- 현재 최고가보다 높을 때만 업데이트하는 Lua 스크립트
local current = redis.call('GET', KEYS[1])
if current = false or tonumber(ARGV[1]) > tonumber(current) then
	redis.call('SET', KEYS[1], ARGV[1])
	return 1   -- 업데이트 성공
else
	return 0   -- 실패 (더 낮은 금액)
end
```

```java
DefaultRedisScript<Long> script = new DefaultRedisScript<>();
script.setScriptText("""
	local current = redis.call('GET', KEYS[1])
	if current = false or tonumber(ARGV[1]) > tonumber(current) then
		redis.call('SET', KEYS[1], ARGV[1])
		return 1   -- 업데이트 성공
	else
		return 0   -- 실패 (더 낮은 금액)
	end
	""");
script.setResultType(Long.class);

Long result = stringRedisTemplate.execute(
	script,
	List.of("auction:room:" + roomId + ":highestBid"),
	String.valueOf(bidAmount)
);

if (result == 0L) throw new BidAmountInvalidException();
```

Lua 스크립트는 원자적이지만 **입찰 이력 DB 저장, WebSocket 발송 등**과 묶을 수 없음
"Redis 업데이트 + DB 저장 + 알림 발송"을 하나의 원자 단위로 만들려면 분산 락이 필요하다.

### 8-3. 분산 락 (Redisson)

분산 락은 여러 서버 인스턴스가 공유하는 **논리적 락** 이다.

```
서버 A ─┐
         ├─▶ Redis: "lock:auction:room:1" 키 선점 (SET NX EX)
서버 B ─┘   → 하나만 성공, 나머지는 대기

선점 성공한 서버만 임계 구역 실행:
	1. Redis 최고가 읽기
	2. 검증
	3. Redis 최고가 업데이트
	4. DB 저장
	5. WebSocket 발송
	   
완료 후 락 키 삭제 → 다음 서버가 락 획득
```

```java
RLock lock = redissonClient.getLock("lock:auction:room:" + roomId);
try {
	lock.tryLock(3, 10, TimeUnit.SECONDS);
	// 임계 구역 - 한 번에 하나의 서버만 실행
} finally {
	lock.unlock();
}
```

Redisson이 내부적으로 Lua 스크립트 + Pub/Sub으로 안전하게 구현한다.

## 9. 해결책 계층 비교 — 언제 어느 레벨을 써야 하나

### 해결책 결정 트리

```
동시성 문제가 있다
    │
    ├─ 단일 서버인가?
    │   ├─ YES → synchronized / AtomicLong 사용 가능
    │   │         단, DB 정합성도 필요하면 DB 락 추가
    │   └─ NO  ↓
    │
    ├─ 서버가 여러 대인가? (분산 환경)
    │   │
    │   ├─ 단순 카운터 증가?
    │   │   └─ Redis INCR (원자 커맨드로 충분)
    │   │
    │   ├─ 조건부 업데이트 (단일 Redis 연산)?
    │   │   └─ Lua 스크립트
    │   │
    │   └─ 여러 단계를 원자적 처리 (Redis + DB + 알림)?
    │       └─ 분산 락 (Redisson) ← 경매 입찰의 경우
    │
    └─ DB 정합성이 최우선, Redis 없이?
        ├─ 충돌이 드묾 → 낙관적 락 (@Version)
        └─ 충돌이 잦음 → 비관적 락 (FOR UPDATE)
```

### 종합 비교표

| 해결책 | 적용 범위 | 성능 | 구현 난이도 | 분산 환경 |
|---|---|---|---|---|
| `synchronized` | JVM 내 | 낮음 (블로킹) | 쉬움 | X |
| `volatile` | JVM 내 가시성만 | 높음 | 쉬움 | X |
| `AtomicLong` | JVM 내 단순 연산 | 높음 | 보통 | X |
| 낙관적 락 (`@Version`) | 단일 DB | 충돌 적으면 높음 | 보통 | X |
| 비관적 락 (`FOR UPDATE`) | 단일 DB | 보통 | 보통 | X |
| Redis 원자 커맨드 | 단순 연산 | 매우 높음 | 쉬움 | O |
| Lua 스크립트 | 단일 Redis 연산 | 높음 | 어려움 | O |
| **Redisson 분산 락** | **복합 연산 전체** | **보통** | **보통** | **O** |

### 이 프로젝트가 Redisson을 선택한 이유

```
요구사항:
  ✓ 서버 2대 이상 (분산 환경)
  ✓ Redis 조회 + 검증 + Redis 업데이트 + DB 저장 + WebSocket 발송
    → 5단계를 하나의 원자 단위로 보호해야 함
  ✓ 성능: DB FOR UPDATE보다 빠르게
  ✓ Deadlock 방지: leaseTime으로 자동 해제
  ✓ 공정성: waitTime으로 무한 대기 방지

결론: Redisson 분산 락이 가장 적합
```
