
## 1. 입찰 트래픽 - 왜 일반 CRUD와 다른가
### 1-1. 일반 서비스 트래픽 패턴
```
일반 게시판: 쓰기 1 : 읽기 10
일반 커머스: 상품 조회 多, 주문 少
```
대부분의 서비스는 읽기가 압도적으로 많고 쓰기는 드물다.
RDB의 Buffer Pool이나 Query Cache가 이런 패턴에서 효율적이다.

### 1-2. 경매 서비스 트래픽 패턴
```
경매 마감 1분 전:
	- 50명이 동시에 "현재 최고가" 조회
	- 10명이 동시에 새 입찰가 제출
	→ 초당 수백 건의 읽기 + 쓰기 혼합됨
```

**특징 3가지**
1. **핫스팟 집중** - 인기 경매방 하나에 트래픽이 몰린다. 전체 경매방이 100개라도 상위 3개에 95%가 집중될 수 있다.
2. **읽기와 쓰기가 같은 데이터를 동시에 건드림** - 최고가 데이터는 읽히는 동시에 업데이트 된다.
3. **정확성이 중요함** - 최고가 메세지는 정확해야 함. 낡은 데이터로 검증하면 동일 금액 입찰이 통과될 수 있음.

## 2. DB만 쓰는 경우
### 2-1. 순수 DB 방식의 처리 흐름
```sql
-- 1. 현재 최고가 조회 (SELECT)
SELECT highest_bid FROM auction_rooms WHERE id = 1 FOR UPDATE;

-- 2. 입찰가 검증
-- (애플리케이션에서 비교)

-- 3. 최고가 업데이트 (UPDATE)
UPDATE auction_rooms SET highest_bid = 150000, highest_bidder_id = 42 WHERE id = 1;

-- 4. 입찰 이력 저장 (INSERT)
INSERT INTO bids (room_id, user_id, amount, created_at)
VALUES (1, 42, 150000, NOW());
```

`FOR UPDATE`로 Row Lock을 걸지 않으면 Race Condition이 발생하고,
걸면 이 쿼리들이 **직렬화**되어 동시 처리가 안된다.

```
🚨 Race Condition

여러 스레드/트랜잭션이 동시에 같은 데이터를 읽고 수정하려고 해서,
실행 순서에 따라 결과가 달라지는 현상.
```

### 2-2. 병목 지점 분석
```
클라이언트 요청
	↓
Spring MVC 스레드 (Tomcat 기본 200개)
	↓
HikariCP 커넥션 획득 (기본 10개) ← ① 커넥션 풀 병목
	↓
MySQL 커넥션
	↓
auction_rooms 테이블 Row Lock 획득 ← ② 락 경합
	↓
디스크 or Buffer Pool에서 페이지 읽기 ← ③ I/O 비용
	↓
업데이트 후 커넥션 반환
```

입찰 요청이 동시에 50개가 들어오면:
- HikariCP 커넥션은 10개 → 40개 요청이 **대기 큐에서 블로킹**
- 커넥션을 얻어도 Row Lock 때문에 **직렬 처리**
- 응답 지연 → 클라이언트 타임아웃 → 재요청 → 더 큰 부하

### 2-3. 실제 DB 쿼리 실행 비용

MySQL에서 `SELECt * FROM auction_rooms WHERE id = 1`이 실행될 때:
```
1. SQL 파싱 & 쿼리 캐시 확인     ~0.01ms
2. 실행 계획 수립 (Optimizer)    ~0.05ms
3. B-Tree 인덱스 탐색           ~0.1ms
4. Buffer Pool에서 페이지 탐색
   - 캐시 히트 시               ~0.1ms
   - 캐시 미스 → 디스크 읽기     ~1~10ms
5. 네트워크 전송 (localhost라도) ~0.05ms

─────────────────────────────────

합계 (Buffer Pool 히트)         ~0.3ms
합계 (디스크 읽기)               ~10ms+
```

Redis는:
```
1. 키 해시 계산 + 해시테이블 탐색    ~0.01ms
2. 메모리에서 값 읽기               ~0.05ms
3. 네트워크 전송 (localhost)       ~0.05ms

─────────────────────────────────

합계                              ~0.1ms (MySQL 히트의 1/3)
```

- 별 차이 없어 보이지만, **동시 요청이 많아질 수록** 차이가 극명하다.

## 3. DB 조회 vs Redis 조회 - 단계별 비용 분석

### 3-1. MySQL의 동시 처리 한계
```
동시 입찰 요청: 100개
HikariCP 커넥션: 10개

타임라인:
T=0ms  : 요청 1~10이 커넥션 획득
T=0ms  : 요청 11~100은 커넥션 대기 시작
T=5ms  : 요청 1~10 처리 완료, 커넥션 반환
T=5ms  : 요청 11~20이 커넥션 획득
...
T=45ms : 요청 91~100이 커넥션 획득
T=50ms : 마지막 요청 처리 완료

→ 100번째 요청의 응답 시간: 50ms (1개 처리 시간의 10배) 
```

### 3-2. Redis의 동시 처리
```
동시 요청: 100개 (Lettuce가 공유 커넥션으로 처리)

타임라인:
T=0ms   : 커맨드 100개를 Lettuce가 파이프라인으로 전송
T=0.5ms : Redis 서버가 싱글스레드로 100개 순차 처리
T=1ms   : 응답 100개 반환

→ 100번째 요청의 응답 시간: ~1ms
```

Lettuce는 **하나의 TCP 커넥션**으로 여러 스레드의 요청을 멀티플렉싱한다.
커넥션 풀이 필요 없고, 커넥션 획득 대기가 없다.

### 3-3. 두 방식의 처리량(Throughput) 비교

| 항목 | MySQL (커넥션 10개) | Redis (Lettuce) |
|---|---|---|
| 초당 처리 가능 요청 | ~1,000 QPS | ~100,000 QPS |
| 동시 요청 100개 시 꼬리 지연 | ~50ms | ~1ms |
| 커넥션 고갈 가능성 | 있음 | 없음 (공유 커넥션) |
| CPU 사용률 | Lock/B-Tree로 높음 | 낮음 |
> **단, 이건 읽기 조회 기준**
> 경매 최고가 **쓰기(업데이트)**는 Redis도 동시성 제어가 필요하다. 
> (Redisson Lock)

## 4. 캐싱 전략 3가지

DB와 Redis를 함께 쓸 때 데이터를 어떻게 동기화할지 전략이 필요함

### 4-1. Chach-Aside (Lazy Loading)
애플리케이션이 **직접 캐시를 관리**함. 가장 일반적인 패턴

```
읽기 흐름:
┌─────────────┐
│ 애플리케이션 │
└──────┬──────┘
       │ ① 캐시 조회
       ▼
┌─────────────┐
│    Redis    │ ─── 캐시 히트 → 데이터 반환
└──────┬──────┘
       │ 캐시 미스
       ▼
┌─────────────┐
│     DB      │ ─── ② DB 조회
└──────┬──────┘
       │ ③ 결과를 Redis에 저장
       ▼
┌─────────────┐
│    Redis    │ ─── ④ 다음 요청부터 캐시 히트
└─────────────┘
```

```java
public Long getHighestBid(Long roomId) {
	String key = "auction:room:" + roomId + ":highestBid";
	String cached = stringRedisTemplate.opsForValue().get(key);
	
	if (cached != null) {
		return Long.parseLong(cahced);  // 캐시 히트
	}
	
	// 캐시 미스 → DB 조회
	Long highestBid = auctionRoomRepository.findHighestBid(roomId);
	if (highestBid != null) {
		stringRedisTemplate.opsForValue()
			.set(key, String.valueOf(highestBid), Duration.ofHours(2));
	}
	return highestBid;
}
```

**장점**
- 실제로 조회된 데이터만 캐시에 올라감 → 불필요한 메모리 낭비 없음
- Redis 장애 시 DB로 폴백 가능 (서비스 중단 없음)
- 구현 단순
**단점**
- 첫 조회는 반드시 DB를 거침 (Cold Start)
- 캐시 미스 후 여러 요청이 동시에 DB를 조회할 수 있음 → **캐시 스탬피드** (8절 참조)
- 쓰기 후 캐시 무효화를 직접 처리해야 함 → **정합성**책임이 개발자한테 있음

### 4-2. Write-Through
```
쓰기 흐름:
┌─────────────┐
│ 애플리케이션 │
└──────┬──────┘
       │ ① DB 업데이트
       ▼
┌─────────────┐
│     DB      │
└──────┬──────┘
       │ 성공
       ▼
┌─────────────┐
│ 애플리케이션 │ ② Redis도 즉시 업데이트
└──────┬──────┘
       ▼
┌─────────────┐
│    Redis    │
└─────────────┘
```

```java
@Transactional
public void updateHighestBid(Long roomId, Long newBid, Long userId) {
	// DB 업데이트
	auctionRoomRepository.updateHighestBid(roomId, newBid, userId);
	
	// Redis도 즉시 업데이트
	String key = "auction:room:" + roomId + ":highestBid";
	stringRedisTemplate.opsForValue().set(key, String.valueOf(newBid));
}
```

**장점**
- 캐시와 DB가 항상 동기화 → 정합성 높음
- 읽기 시 항상 캐시 히트 (Cold Start 없음)
**단점**
- 한 번도 읽히지 않는 데이터도 캐시에 씀 → 메모리 낭비
- 쓰기 지연 증가 (DB + Redis 둘 다 기다려야 함)
- Redis 장애 시 쓰기 실패 → 가용성 저하
- **`@Transactional` 안에서 Redis 함께 쓰면 롤백 불일치** 문제 발생

### 4-3. Write-Behind (Write-Back)

**캐시에만 먼저 쓰고, DB 반영은 비동기**로 나중에 처리한다.

```
쓰기 흐름:
┌─────────────┐
│ 애플리케이션 │
└──────┬──────┘
       │ ① Redis에만 즉시 저장 (응답 반환)
       ▼
┌─────────────┐
│    Redis    │
└──────┬──────┘
       │ ② 배치/큐로 비동기 DB 반영
       ▼ (나중에)
┌─────────────┐
│     DB      │
└─────────────┘
```

**장점**
- 쓰기 속도 극대화 (Redis에만 쓰면 즉시 반환)
- DB 쓰기를 배치로 묶어서 처리 → DB 부하 최소화

**단점**
- Redis 장애 시 아직 DB 반영 안 된 데이터는 **영구 손실**
- 구현 복잡 (비동기 처리, 실패 재시도, 순서 보장 필요)

### 전략 비교 요약
| 항목          | Cache-Aside | Write-Through | Write-Behind |
| ----------- | ----------- | ------------- | ------------ |
| 읽기 성능       | 첫 요청만 DB    | 항상 캐시         | 항상 캐시        |
| 쓰기 성능       | 빠름          | 보통            | 최고           |
| 정합성         | 개발자 책임      | 높음            | 낮음 (지연)      |
| 데이터 손실 위험   | 없음          | 없음            | **있음**       |
| Redis 장애 영향 | 읽기만 DB로     | 쓰기 실패         | 데이터 손실       |
| 구현 난이도      | 쉬움          | 보통            | 어려움          |

## 5. BidFlow 프로젝트에 Cache-Aside 쓴 이유
### 요구사항 분석

| 요구사항 | 중요도 |
|---|---|
| 최고가 조회 성능 | 최고 |
| 데이터 정확성 | 최고 |
| 서비스 가용성 (Redis 장애 시) | 높음 |
| 구현 복잡도 최소화 | 보통 |

### Write-Through를 쓰지 않는 이유

```java
// 이렇게 하면 문제가 있다
@Transactional
public void placeBid(Long roomId, Long amount) {
    auctionRoomRepository.updateHighestBid(roomId, amount);  // DB 업데이트
    redisTemplate.opsForValue().set(key, amount);            // Redis 업데이트

    // DB 커밋 전에 Redis는 이미 업데이트됨!
    // DB가 롤백되면? Redis는 이미 새 값을 갖고 있음 → 불일치
}
```

JPA `@Transactional`은 커밋 전에는 DB에 반영되지 않는다.  
그런데 Redis는 `@Transactional` 밖에 있어서 **즉시 반영**된다.  
DB 롤백 시 Redis를 수동으로 되돌려야 하는 복잡성이 생긴다.

### Write-Behind를 쓰지 않는 이유

경매 최고가는 **"낙찰" 결정에 사용**된다.  
비동기로 DB에 늦게 반영된다면, Redis 장애 시 최종 낙찰 금액이 사라질 수 있다.  
경매는 금전이 오가므로 데이터 손실은 허용 불가다.

### Cache-Aside + Redis를 정합성 기준으로 쓰는 변형

이 프로젝트는 일반 Cache-Aside와 약간 다르다.

```
일반 Cache-Aside: DB가 정합성의 기준 (Source of Truth)
이 프로젝트:      Redis가 최고가의 기준, DB는 이력 저장소
```

```
입찰 처리:
┌────────────────────────────────────┐
│ 1. Redisson Lock 획득              │
│ 2. Redis에서 현재 최고가 조회      │ ← 정합성 기준
│ 3. 새 입찰가 > 최고가 검증        │
│ 4. Redis에 새 최고가 저장          │ ← 즉시 업데이트
│ 5. DB에 입찰 이력 INSERT          │ ← 비동기 or 동기
│ 6. Lock 해제                       │
└────────────────────────────────────┘
```

DB `auction_rooms.highest_bid` 컬럼은 조회 성능을 위한 컬럼이 아니라  
**감사(Audit) 목적**이다. 실시간 검증은 Redis가 한다.

## 6. 캐시 정합성 문제 - 언제 DB와 Redis가 달라지는가
### 시나리오 1. 서버 재시작
```
상황: 경매 진행 중 서버 재시작
→ Redis 연결은 유지되지만, 애플리케이션 메모리는 초기화됨
→ Redis 데이터는 그대로 (별도 프로세스)
→ 정합성 유지됨 (OK)
```

### 시나리오 2. Redis 재시작 (No Persistence 설정)
```
상황: Redis 프로세스 재시작
→ Redis 데이터 전체 소실
→ 캐시 미스 발생 → DB에서 재로딩
→ 단, 마지막 DB 반영 후 Redis에서 일어난 업데이트가 있었다면? >> 달라짐
```

BidFlow 프로젝트는 입찰 시 **DB에도 즉시 저장**하므로 Redis 소실 시 DB가 최신 상태이다.
Cache-Aside로 DB에서 재로딩하면 복구된다. (아래 플로우)
```java
// getHighestBid()의 캐시 미스 처리가 자동으로 복구 역할을 함
if (cached == null) {
	// DB에서 최신값 로드 → Redis 재적재
	Long highestBid = auctionRoomRepository.findHighestBid(roomId);
	stringRedisTemplate.opsForValue().set(key, String.valueOf(highestBid));
}
```

### 시나리오 3: DB 롤백 후 Redis는 업데이트 된 상태
```java
@Transactional
public void placeBid(Long roomId, Long userId, Long bidAmount) {
	// Redis 먼저 업데이트 (Lock 안)
	stringRedisTemplate.opsForValue().set(key, String.valueOf(bidAmount));
	
	// DB 저장
	bidRepository.save(bid);
	
	// 만약 여기서 예외 발생 → DB 롤백
	// → Redis는 이미 업데이트 됨 → 불일치
	throw new RuntimeException("DB 오류");
}
```

**해결책: 보상 트랜잭션 or 이벤트 기반 처리**
```java
// 방법 1 : TransactionSynchronizationManager로 커밋 후 Redis 업데이트
TransactionSynchronizationManager.registerSynchronization(
	new TransactionSynchronizationAdapter() {
		@Override
		public void afterCommit() {
			stringRedisTemplate.opsForValue().set(key, String.valueOf(bidAmount));
		}
	}
);
```

```java
// 방법 2 : DB 저장 후 Redis 업데이트 (순서를 DB 먼저 처리)
bidRepository.save(bid);        // DB 먼저 (트랜잭션 안)

// @Transactional 종료 → 커밋 완료
stringRedisTemplate.opsForValue().set(key, ...); // 커밋 후 Redis
```

BidFlow 프로젝트는 Lock 안에서 Redis를 정합성 기준으로 쓰기 때문에,
**DB 롤백이 발생하면 Redis 값을 원상 복구하는 로직이 반드시 필요**하다.

### 시나리오 4: 캐시 무효화 타이밍

```
입찰자 A: Redis 최고가 = 100,000원 읽음
입찰자 B: 입찰 성공 → Redis = 110,000원
입찰자 A: (110,000원 모르는 채로) 105,000원 입찰 시도
→ Lock 안에서 Redis 다시 읽으면 110,000원 → 검증 실패 → 재시도 안내
```

Lock 안에서 Redis를 읽기 때문에 이 시나리오는 정상적으로 처리됨.

## 7. 캐시 스탬피드 - 동시에 캐시 미스가 나면?

### 문제 상황

경매 시작 직후 또는 Redis 재시작 직후:

```
T=0: Redis에 "auction:room:1:highestBid" 키 없음
T=0: 100개의 요청이 동시에 캐시 미스
T=0: 100개의 요청이 모두 DB로 직행
→ DB에 100개의 동시 쿼리 → DB 과부하
→ DB가 느려지면 더 많은 요청이 쌓임 → 연쇄 장애 가능
```

이를 **Thunder Herd Problem** 또는 **Cache Stampede**라고 한다.


### 해결책 1: Mutex Lock (권고)

첫번째 요청만 DB 조회하고, 나머지는 기다린 후 캐시 읽음

```java
public Long getHighestBidSafe(Long roomId) {
	String cacheKey = "auction:room:" + roomId + ":highestBid";
	String lockKey = "lock:cache:load:" + roomId;
	
	String cached = stringRedisTemplate.opsForValue().get(cacheKey);
	if (cached != null) return Long.parseLong(cached);
	
	// 캐시 미스 - 락을 통해 DB 조회를 직렬화
	RLock lock = redissonClient.getLock(lockKey);
	try {
		lock.tryLock(2, 5, TimeUnit.SECONDS);
		
		// 락 획득 후 캐시 재확인 (Double-Checked Locking)
		cached = stringRedisTemplate.opsForValue().get(cacheKey);
		if (cahced != null) return Long.parseLong(cached); // 다른 스레드가 이미 로딩
		
		// 진짜 DB 조회 (직렬화됨 - 한 번만 실행)
		Long highestBid = auctionRoomRepository.findHighestBid(roomId);
		stringRedisTemplate.opsForValue()
			.set(cacheKey, String.valueOf(highestBid), Duration.ofHours(2));
		return highestBid;
	} finally {
		if (lock.isHeldByCurrentThread()) lock.unlock();
	}
}
```

**Double-Checked Locking** 패턴 : 락 획득 후 캐시를 한 번 더 확인해서, 락 대기 중에 다른 스레드가 이미 캐시를 채웠으면 DB 조회를 건너 뜀


### 해결책 2: 캐시 워밍 (Cache Warming)

서버 시작 시 또는 경매 시작 시 미리 Redis 에 데이터 로딩함

```java
@EventListener(ApplicationReadyEvent.class)
public void warmUpCache() {
	List<AuctionRoom> activeRooms = auctionRoomRepository.findAllActive();
	for (AuctionRoom room : activeRooms) {
		String key = String.format(HIGHEST_BID_KEY, room.getId());
		stringRedisTemplate.opsForValue().set(
			key,
			String.valueOf(room.getHighestBid()),
			Duration.between(LocalDateTime.now(), room.getEndTime())
		)
	}
}
```


## 9. TTL 설계 - 경매 종료 시각과 맞추기

### 9-1. TTL이란?
- **TTL(Time To Live)** 은 Redis 키에 설정하는 **만료 시간** 이다.
- 설정된 시간이 지나면 Redis가 해당 키를 **자동으로 삭제** 한다.
### 9-2. TTL 내부 동작 원리
- Redis는 TTL을 관리하기 위해 두 가지 방식을 **병행**한다.

#### 9-2-1. Lazy Expiration (지연 삭제)

키에 접근하는 순간 만료 여부를 확인하고, 만료됐으면 그 자리에서 삭제
```
GET auction:room:1:highestBid
	→ Redis 내부: "이 키 TTL 확인"
	→ TTL 초과 → 키 삭제 → nil 반환
```

**장점**: 만료된 키를 굳이 미리 찾지 않아도 됨
**단점**: 오랫동안 아무도 접근하지 않는 키는 만료 후에도 메모리를 점유하고 있음
#### 9-2-2. Active Expiration (능동 삭제)

Redis는 **100ms마다** 백그라운드 작업을 실행 해 만료된 키를 샘플링해서 삭제함
```
매 100ms마다:
	1. 각 DB(0~15번)에서 TTL 있는 키를 20개 무작위 샘플링
	2. 만료된 키 삭제
	3. 삭제된 키가 25% 이상이면 → 즉시 다음 라운드 반복
	4. 25% 미만이면 → 다음 100ms까지 대기
```

전부 스캔하지 않고 샘플링 하는 이유: 전체 스캔은 Redis를 블로킹하기 때문.
그래서 만료 직후 즉시 메모리가 회수되지 않을 수 있음

> **실전 포인트**: TTL이 지난 키는 `GET`으로 접근해도 nil이 반환되지만,  
> 메모리 사용량(`INFO memory`)에서는 아직 잡힐 수 있다.  
> `OBJECT FREQ` 나 `DEBUG SLEEP` 없이 확인하려면 `TTL key`가 -2이면 삭제 완료다.

### 9-3. TTL 관련 커맨드 총정리
```bash
#TTL 설정 방법들
SET key value EX 3600               # 초 단위로 저장과 동시에 설정
SET key value PX 3600000            # 밀리초 단위
SET key value EXAT 1750000000       # 유닉스 타임스탬프(초)로 만료 시각 지정
SET key value PXAT 1750000000000    # 유닉스 타임스탬프(밀리초)로 지정

EXPIRE key 3600             # 이미 있는 키에 초 단위 TTL 설정
PEXPIRE key 3600000         # 이미 있는 키에 밀리초 단위 TTL 설정
EXPIREAT key 1750000000     # 유닉스 타임스탬프(초)로 만료 시각 지정
PEXPIREAT key 1750000000000 # 유닉스 타임스탬프(밀리초)로 만료 시각 지정

# TTL 조회
TTL key   # 남은 초 반환 / -1: TTL 없음 / -2: 키 없음(삭제됨)
PTTL key  # 남은 밀리초 반환

# TTL 제거 (영구 보관으로 전환)
PERSIST key  # TTL 제거 → -1 상태로 변환

# EXPIRE 옵션 (Redis 7.0+)
EXPIRE key 3600 NX   # TTL이 없는 키에만 설정
EXPIRE key 3600 XX   # TTL이 이미 있는 키에만 설정
EXPIRE key 3600 GT   # 현재 TTL보다 클 때만 설정 (연장만 가능)
EXPIRE key 3600 LT   # 현재 TTL보다 작을 때만 설정 (단축만 가능)
```

### 9-4. TTL 반환값의 의미

| TTL 반환값 | 의미 |
|---|---|
| **양수** | 남은 초(또는 밀리초) |
| **-1** | 키는 존재하지만 TTL 없음 (영구 보관) |
| **-2** | 키 자체가 없음 (삭제됐거나 처음부터 없음) |

```java
Long ttl = stringRedisTemplate.getExpire("auction:room:1:highestBid",
TimeUnit.SECONDS);

if (ttl == null || ttl == -2L) {
	// 키 없음 → 캐시 미스
} else if (ttl == -1L) {
	// TTL 없음 → 의도치 않은 영구 저장 → 버그 가능성 체크
} else {
	// 정상 - ttl초 후 만료
}
```

#### 9-5. SET과 EXPIRE의 원자성 주의

```java
// 위험 - 두 커맨드 사이에 서버 장애 나면 TTL 없는 키가 남음
stringRedisTemplate.opsForValue().set(key, value);
stringRedisTEmplate.expire(key, Duration.ofHours(2));

// 안전 - 단일 커맨드로 저장 + TTL 동시 설정
stringRedisTemplate.opsForValue().set(key, value, Duration.ofHours(2));
```

`SET key value EX 3600`은 단일 커맨드로, 저장과 TTL을 동시에 설정한다.
반면 `SET` 후 `EXPIRE`는 두 커맨드 사이에 장애가 나면 TTL 없는 키가 영구적으로 남는다.

#### 9-6. 왜 TTL이 중요한가

```
경매방 1: 오후 3시 종료
Redis 키: "auction:room:1:highestBid"

TTL 없으면:
	- 경매 종료 후에도 키가 남아있음
	- 신규 경매방이 같은 ID를 재사용하면 이전 최고가가 보임
	- 메모리 낭비
	  
TTL = 경매 종료 시각:
	- 경매 종료와 함께 자동 삭제
	- 이후 조회는 DB로 폴백
```

#### 9-7. TTL 계산 로직

```java
public void cacheHighestBid(Long roomId, Long amount, LocalDateTime endTime) {
	String key = String.format(HIGHEST_BID_KEY, roomId);
	long ttlSeconds = ChronoUnit.SECONDS.between(LocalDateTime.now(), endTime);
	
	if (ttlSeconds <= 0) {
		// 이미 종료된 경매 - 캐시 저장 안 함
		return;
	}
	
	// 약간의 여유를 두어 종료 직후에도 조회 가능하게 (선택사항)
	long bufferSeconds = 60;
	stringRedisTemplate.opsForValue()
		.set(key, String.valueOf(amount), Duration.ofSeconds(ttlSeconds + bufferSeconds));
}
```

@RequiredArgsConstructor
public class PerformanceTest {

		for (int i = 0; i < iterations; i++) {

		long redisStart = System.nanoTime();

	}
}
