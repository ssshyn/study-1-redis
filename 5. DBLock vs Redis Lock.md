
## 2. 동시성 테스트 도구 이해 - CountDownLatch / ExecutorService / AtomicInteger

BidFlow 프로젝트 > `AuctionRoomBidConcurrencyTest` 테스트 문서

### 2-1. CountDownLatch - 스레드 동기화

`CountDownLatch`는 하나 이상의 스레드가 **다른 스레드들의 완료를 기다리게**하는 동기화 도구

```java
CountDownLatch latch = new CountDownLatch(3);  // 카운트: 3

// 스레드 A
latch.countDown();  // 카운트 : 2

// 스레드 B
latch.countDown();  // 카운트 : 1

// 스레드 C
latch.countDown();  // 카운트 0 → 대기 중인 스레드 깨어남

// 메인 스레드
latch.await();  // 카운트가 0이 될 때까지 블로킹
```

#### 2-1-1. 동시성 테스트에서의 활용 패턴
```
문제: submit()으로 스레드를 제출하면 각자 다른 시점에 실행 시작
	→ 진짜 동시 실행이 아니라 순차 실행에 가까워짐
	→ Race Condition 재현이 어려움
	
해결: 두 개의 CountDownLatch로 "동시 출발 + 전체 완료 대기"
```

```java
CountDownLatch startLatch = new CountDownLatch(1);  // 출발 신호
CountDownLatch doneLatch = new CountDownLatch(100); // 완료 대기

for (int i=0; i<100; i++) {
	executor.submit(() -> {
		startLatch.await(); // 출발 신호 대기
		
		doWork();
		doneLatch.countDown();  // 완료 신호
	});
}

// 모든 스레드가 '출발 신호 대기'에서 대기 중임
startLatch.countDown();  // 동시 출발
doneLatch.await();       // 100개 모두 완료까지 대기
```

### 2-2. ExecutorService - 스레드 풀 관리

```java
// 고정 크기 스레드 풀
ExecutorService executor = Executors.newFixedThreadPool(32);

// 작업 제출
Future<Long> future = executor.submit(() -> {
	return placeBid(roomId, userId, amount);
});

// 완료 대기 + 결과 수집
Long result = future.get();

// 풀 종료 (새 작업 거부 + 기존 작업 완료 대기)
executor.shutdown();
executor.awaitTermination(30, TimeUnit.SECONDS);
```

### 2-2-1. 스레드 풀 크기와 동시성 테스트

```
테스트 목표 : 동시 입찰 100개

스레드 풀 크기 = 100: 진짜 동시 실행 (최대 Race Condition 재현)
스레드 풀 크기 = 10: 10개씩 배치 실행 (부분적 동시성)
스레드 풀 크기 = 1: 순차 실행 (동시성 없음)

→ Race Condition 재현 목적이면 스레드 수 = 요청 수로 설정
```

### 2-3. AtomicInteger - 스레드 안전 카운터

```java
AtomicInteger successCount = new AtomicInteger(0);
AtomicInteger failCount = new AtomicInteger(0);

// 스레드에서
successCount.incrementAndGet();  // 원자적 + 1
failCount.incrementAntGet();

// 메인 스레드에서 결과 확인
System.out.println("성공: " + successCount.get());
System.out.println("실패: " + failCount.get());
```
일반 `int` 카운터를 여러 스레드에서 `++`하면 Lost Update 발생
`AtomicInteger`는 CAS로 원자적 증가를 보장

```java
@Test
@DisplayName("서로 다른 금액으로 동시 입찰 > 최고가만 남아야 함")
void differentAmountConcurrentBid() throws InterruptedException {
	Long roomId = 1L;
	int threadCount = 100;
	
	CountDownLatch startLatch = new CountDownLatch(1);
	CountDownLatch doneLatch = new CountDownLatch(threadCount);
	AtomicInteger successCount = new AtomicInteger(0);
	AtomicInteger failCount = new AtomicInteger(0);
	List<Long> successAmounts = Collections.synchronizedList(new ArrayList<>());
	
	for (int i=1; i <= threadCount; i++) {
		final long bidAmount = 100_000L + (i * 1_000L); // 101,000 ~ 200,000
		
		executorService.submit(() -> {
			try {
				startLatch.await();  // 동시 출발 대기
				
				auctionRoomService.placeBid(roomId, (long)(Math.random() * 1000), bidAmount);
				successCount.incrementAndGet();
				successAmounts.add(bidAmount);
			} catch (BidAmountInvalidException e) {
				failCount.incrementAndGet(); // 더 낮은 금액 → 정상 실패
			} ctach (BidFailedException e) {
				failCount.incrementAndGet(); // 락 획득 실패 → 재시도 필요
			} catch (InterruptedException e) {
				Thread.currentThread().interrupt();
			} finally {
				doneLatch.countDown();
			}
		});
	}
	
	startLatch.countDown();  // 동시 출발
	doneLatch.await();
	
	// 검증
	String finalBid = stringRedisTemplate.opsForValue().get("auction:room:" + roomId + ":highestBid");
	long finalHighest = Long.parseLong(finalBid);
	
	System.out.println("성공: " + successCount.get());
	System.out.println("실패: " + failCount.get());
	System.out.println("최종 최고가: " + finalHighest);
	System.out.println("DB 입찰 건수: " + bidRepository.countByAuctionRoomId(roomId));
	
	// 최종 최고가는 200,000 이어야 한다.
	assertThat(finalHighest).isEqualTo(200_200L);
	
	// DB 입찰 건수 = 성공 횟수 (일치해야 정합성 OK)
	assertThat(bidRepository.countByAuctionRoomId(roomId))
	.isEqualTo(successCount.get());
}
```

#### 테스트 의도

```
정상 동작 시 :
	- 최종 최고가 = 200,000 (제일 높은 금액)
	- DB 입찰 건수 = Redis 기준 성공 횟수 (정합성 검증)

분산 락이 없을 때:
	- 최종 최고가 = 마지막에 저장된 값(타이밍에 따라 랜덤)
	- 실행마다 다른 결과 → Race Condition 확인
```

### Case 3: 동일 금액으로 동시 입찰
```java
@Test
@DisplayName("동일 금액으로 동시 입찰 > 한 명만 성공")
void sameAmountConcurrentBid() throws InterruptedException {
	Long roomId = 1L;
	long sameAmount = 150_000L;  // 모두 같은 금액
	
	CountDownLatch startLatch = new CountDownLatch(1);
	CountDownLatch doneLatch = new CountDownLatch(THREAD_COUNT);
	AtomicInteger successCount = new AtomicInteger(0);
	
	for (int i=0; i < THREAD_COUNT; i++) {
		final long userId = i + 1;
		
		executorService.submit(() -> {
			try {
				startLatch.await();
				auctionRoomService.placeBid(roomId, userId, sameAmount);
				successCount.incrementAndGet();
			} catch (Exception e) {
				// 실패는 정상
			} finally {
				doneLatch.countDown();
			}
		});
	}
	
	startLatch.countDown();
	doneLatch.await();
	
	// 동일 금액이어도 한 명만 성공해야 함
	// (현재 최고가가 100,000 / 150,000은 처음엔 통과 이후는 <= 150,000 으로 실패)
	assertThat(successCount.get()).isEqualTo(1);
	
	String finalBid = stringRedisTemplate.opsForValue().get("auction:room:" + roomId + ":highestBid");
	
	assertThat(Long.parseLong(finalBid)).isEqualTo(sameAmount);
	
	// DB에도 정확히 1건만 저장되어야 함
	assertThat(bidRepository.countByAuctionRoomId(roomId)).isEqualTo(1);
}
```

#### 동일 금액
```
초기 최고가: 100,000
입찰가: 150,000 (100명 동시)

락 있는 경우:
	스레드 1: 락 획득 → 읽기(100,000) → 150,000 > 100,000 → 저장(150,000) → 락 해제
	스레드 2: 락 획득 → 읽기(150,000) → 150,000 <= 150,000 → 실패 스레드 3~100: 모두 실패
	
성공: 1명, 실패 99명 (O)

락 없는 경우:
	100개 스레드 모두 읽기(100,000) → 150,000 > 100,000 → 통과
	→ 100명 모두 성공 처리 → DB에 100건 INSERT
	→ 동일 금액 낙찰자가 100명 생기는 버그
```

## 4. DB Lock 방식 - 낙관적 락 / 비관적 락 성능

### 4-1. 낙관적 락 테스트

```java
@Test
@DisplayName("낙관적 락 - 동시 입찰 성능 테스트")
void optimisticLockConcurrencyTest() throws InterruptedException {
	int threadCount = 100;
	CountDownLatch startLatch = new CountDownLatch(1);
	CountDownLatch doneLatch = new CountDownLatch(threadCount);
	
	AtomicInteger successCount = new AtomicInteger(0);
	AtomicInteger retryCount = new AtomicInteger(0);
	AtomicInteger failCount = new AtomicInteger(0);
	
	long startTime = System.currentTimeMIllis();
	
	for (int i=1; i<=threadCount; i++) {
		final long bidAmount = 100_000L + (i * 1_000L);
		
		executorService.submit(() -> {
			try {
				startLatch.await();
				
				int maxRetry = 5;
				for (int attempt = 0; attempt < maxRetry; attempt++) {
					try {
						auctionRoomServiceWithOptimisticLock.placeBid(roomId, userId, bidAmount);
						successCount.incrementAndGet();
						break;
					} catch (OptimisticLockingFailureException e) {
						retryCount.incrementAndGet();
						if (attempt == maxRetry - 1) {
							failCount.incrementAndGet();
						}
						Thread.sleep(10 * (long) Math.pow(2, attempt)); //지수 백오프
					}
				} catch (Exception e) {
					failCount.incrementAndGet();
				} finally {
					doneLatch.countDown();
				}
			}
		});
	}
	
	startLatch.countDown();
	doneLatch.await();
	
	long elapsed = System.currentTimeMillis() - startTime;
	
	System.out.println("=== 낙관적 락 결과 ===");
    System.out.println("소요 시간: " + elapsed + "ms");
    System.out.println("성공: " + successCount.get());
    System.out.println("재시도 총 횟수: " + retryCount.get());
    System.out.println("최종 실패: " + failCount.get());
}
```

#### 낙관적 락 문제 - 재시도 폭발

```
동시 입찰 100개:
	1라운드: 1개 성공, 99개 OptimisticLockException → 재시도
	2라운드: 1개 성공, 98개 재시도
	...
	100라운드: 마지막 1개 성공
	
총 커밋 시도 횟수: 100 + 99 + 98 + ... + 1 = 5,050번
DB 쿼리 최소 5,050 x 2 = 10,100번 (SELECT + UPDATE)
```

충돌이 잦은 상황에서 낙관적 락은 DB에 엄청난 부하를 줌

## 6. DB Lock vs Redis Lock 비교

### 6-1. 아키텍처 관점

```
DB Lock (비관적 락) 흐름:
Client → Spring → HikariCP → MySQL (FOR UPDATE) → 처리 → 커밋 → 반환

Redis Lock (Redisson) 흐름:
Client → Spring → Redis (tryLock) → MySQL (INSERT) → Redis (unlock) → 반환

추가 Redis 왕복 발생 (락 획득 + 해제)
대신 MySQL Row Lock 경합이 사라짐
```

### 6-2. 9가지 항목 정면 비교

#### 6-2-1. 정합성

| | DB Lock | Redis Lock |
|---|---|---|
| **단일 서버** | 보장 | 보장 |
| **다중 서버** | 보장 (DB 공유) | 보장 (Redis 공유) |
| **Redis 장애 시** | 영향 없음 | 락 획득 불가 → 서비스 중단 |
| **DB 장애 시** | 락 획득 불가 | Redis 락은 잡혔지만 DB 저장 실패 |

#### 6-2-2. 성능 (처리량)

```
비관적 락:
	처리량 = DB 커넥션 수 x (1/평균 처리 시간)
	= 10커넥션 x (1/5ms) = 2,000 TPS (이론)
	실제: 락 대기 + 컨텍스트 스위칭으로 훨씬 낮음
	
Redisson:
	처리량 ≈ Redis 처리 속도 한계
	= ~100,000 TPS (Redis 단일 스레드 한계)
	실제: 임계 구역 직렬화로 그보다 낮지만 DB 병목보다 훨씬 높음
```

#### 6-2-3. 커넥션 소비

```
비관적 락:
	입찰 처리 중 DB 커넥션 점유
	100개 동시 요청 → 10개 커넥션으로 처리
	→ 90개 요청이 커넥션 풀 대기
	
Redisson:
	락 획득/해제: Redis 커넥션 (Lettuce 공유, 거의 무제한)
	DB 커넥션: INSERT 시점에만 짧게 사용
	→ 커넥션 풀 고갈 위험 없음
```

#### 6-2-4. 트랜잭션 범위

```
비관적 락:
	@Transactional 하나로 전체 묶음
	SELECT FOR UPDATE ~ UPDATE ~ INSERT 모두 같은 트랜잭션
	→ DB 수준의 원자성 보장
	
Redisson:
	@Transactional(입찰 이력 INSERT) + Redis 조작을 분리
	→ DB와 Redis 간 원자성 없음
	→ DB 롤백 시 Redis 보상 필요
```

#### 6-2-5. 데드락 위험성

```
DB 비관적 락:
	트랜잭션 A: auction_rooms Row Lock → bids INSERT 대기
	트랜잭션 B: bids INSERT → auction_rooms Row Lock 대기
	→ MySQL 데드락 감지 → 하나를 롤백 + 에러 반환
	
Redisson:
	tryLock(waitTime) → waitTime 초과 시 포기 → 데드락 탈출
	leaseTime → 락 보유자 죽어도 자동 해제
	→ 데드락 구조적 방지
```

#### 6-2-6. 분산 환경 (다중 서버)

```
DB 비관적 락:
	DB가 하나면 분산 환경에서도 동작
	DB가 샤딩되거나 다중 DB면 Row Lock이 분산됨 → 무력화
	
Redisson:
	Redis가 하나면 서버가 몇 대든 락 공유
	Redis 클러스터라면 추가 설계 필요
```

#### 6-2-7. 페일오버 (장애 복구)

```
DB 비관적 락:
	MySQL Replication: Primary 장애 → Secondary 승격 
	이 과정에서 진행 중인 트랜잭션 롤백 → 락 자동 해제
	재시도 시 복구 가능
	
Redisson:
	Redis Sentinel: Primary 장애 → Sentinel이 Replica 승격
	장애 순간 진행 중이던 락은? → 복잡한 엣지 케이스 존재
	leaseTime으로 자동 해제되지만, 이 사이에 다른 서버가 락 획득 가능
	
---
* Redis는 보통 Primary / Replica(복제) 서버로 운영
  Sentinel : 서버 정상 여부 감시하는 프로세스
```

#### 6-2-8. 운영 복잡도

```
DB 비관적 락:
	추가 인프라 없음 (DB만 있으면 됨)
	MySQL FOR UPDATE는 잘 알려진 기술
	모니터링: MySQL slow query log, INFORMATION_SCHEMA.INNODB_LOCKS
	
Redisson:
	Redis 인프라 추가 필요 (고가용성위해 Sentinel/Cluster)
	분산 락 특유의 디버깅 어려움 (락 누수, 소유자 확인)
	모니터링: Redis INFO, TTL 확인, 락 키 잔존 여부
```

#### 6-2-9. 비용

```
DB 비관적 락:
	이미 있는 MySQL 활용 → 추가 비용 없음

Redisson:
	Redis 서버 추가 (EC2 기준 월 수만원 ~ 수십만원)
	Sentinel 구성이면 Redis 3대 (Primary 1 + Replica 2)
```

## 7. 언제 DB Lock / Redis Lock 써야 하는가

### 7-1. 결정 트리

```
동시성 제어가 필요하다
    │
    ├─ 서버가 단일이고, 성능 요구사항이 낮은가?
    │   └─ YES → DB 비관적 락 (단순, 추가 인프라 없음)
    │
    ├─ 충돌 빈도가 낮고, 재시도 비용이 허용되는가?
    │   └─ YES → DB 낙관적 락 (@Version, 충돌 시 재시도)
    │
    ├─ 다중 서버이고, 성능이 중요한가?
    │   └─ YES → Redis 분산 락 (Redisson)
    │
    ├─ DB 트랜잭션 원자성이 최우선인가? (금융, 재고 차감 등)
    │   └─ YES → DB 비관적 락 + 별도 Redis 조합 고려
    │
    └─ 이미 Redis 인프라가 있고, 고성능이 필요한가?
        └─ YES → Redisson 분산 락
```

### 7-2. 케이스별 권고

| 케이스 | 권고 | 이유 |
|---|---|---|
| 스타트업 초기 (서버 1대) | DB 비관적 락 | 단순, Redis 인프라 불필요 |
| 커머스 재고 차감 (적은 충돌) | DB 낙관적 락 | 충돌 드묾, 재시도 허용 |
| **실시간 경매 입찰** | **Redisson 분산 락** | **다중 서버, 고빈도 충돌, 성능 우선** |
| 좌석 예매 (고충돌, 단일 DB) | DB 비관적 락 | 트랜잭션 원자성 최우선 |
| API Rate Limiting | Redis (RSemaphore) | 횟수 제한이므로 DB 불필요 |
| 배치 작업 중복 실행 방지 | Redisson (leaseTime 길게) | 서버 여러 대에서 하나만 실행 |
| 분산 스케줄러 단독 실행 | Redisson + ShedLock | 잘 알려진 패턴 |

### 7-3. 같이 쓰는 패턴 — Redis Lock + DB Lock

두 가지를 같이 쓰는 경우도 있다:

```java
// 외부: Redisson으로 서버 간 경합 줄이기
RLock distributedLock = redissonClient.getLock("lock:auction:room:" + roomId);
distributedLock.tryLock(3, 10, TimeUnit.SECONDS);

    // 내부: DB FOR UPDATE로 최종 정합성 보장 (이중 안전망)
    @Transactional
    public void placeBid(...) {
        AuctionRoom room = repo.findByIdWithLock(roomId); // FOR UPDATE
        // ...
    }

distributedLock.unlock();
```

- Redisson 락: 대부분의 경합을 Redis 레벨에서 차단 → DB 부하 감소
- DB FOR UPDATE: 혹시 모를 Redis 장애 시 최후 안전망

이 패턴은 성능과 정합성을 모두 원할 때 사용하지만,  
구현 복잡도가 올라가므로 정말 필요한지 신중하게 판단해야 한다.

## 8. 경매 시스템 최종 아키텍처 설계 결정

### 이 프로젝트의 선택 정리

```
선택: Redisson 분산 락 (Redis Lock)
이유:
  1. 다중 서버 운영 전제 → Java synchronized / DB Lock으로는 부족
  2. 입찰 = 읽기(Redis) + 쓰기(Redis) + 저장(DB) + 알림(WebSocket)
     → 여러 단계를 하나의 임계 구역으로 보호해야 함
  3. 경매 마감 직전 고빈도 충돌 → 낙관적 락의 재시도 폭발 문제 회피
  4. 이미 Redis 인프라 존재 (캐싱 용도로 사용 중)
     → 추가 비용 없이 Redisson 적용 가능
```

### 최종 입찰 처리 흐름 전체 그림

```
클라이언트 입찰 요청
    │
    ▼
Spring MVC 스레드 (Tomcat)
    │
    ▼ tryLock(waitTime=3s, leaseTime=10s)
Redis ─────────────────────────────────────────┐
  "lock:auction:room:1" 키 선점                │
  실패 시 Pub/Sub 대기 (최대 3초)               │
  성공 시 임계 구역 진입                         │
    │                                           │
    ▼ (임계 구역)                                │
Redis GET "auction:room:1:highestBid"          │  Watchdog X
    │ (캐시 히트 → 최고가 조회)                   │  (leaseTime 명시)
    ▼                                           │
입찰가 검증 (새 입찰가 > 현재 최고가)              │
    │                                           │
    ▼                                           │
Redis SET "auction:room:1:highestBid" = 신규값  │
Redis SET "auction:room:1:highestBidder" = 유저ID│
    │                                           │
    ▼                                           │
MySQL INSERT bids (입찰 이력 저장)              │
    │                                           │
    ▼                                           │
WebSocket 브로드캐스트 (전체 참여자 알림)          │
    │                                           │
    ▼ unlock()                                  │
Redis ─────────────────────────────────────────┘
  "lock:auction:room:1" 키 삭제
  Pub/Sub 채널에 해제 신호 발행
    │
    ▼
응답 반환 (BidResult.success)
```

### 남은 설계 과제 (프로젝트 발전 방향)

```
현재 구현의 한계:
1. Redis 장애 → 입찰 전체 불가 (단일 장애점)
   → Redis Sentinel 구성으로 고가용성 확보

2. DB 롤백 시 Redis 불일치
   → TransactionSynchronizationManager.afterCommit() 으로 커밋 후 Redis 업데이트

3. 경매 종료 처리 자동화 없음
   → Week 6 Day 4: Spring Scheduler / Quartz로 자동 종료

4. 입찰 급증 시 WebSocket 브로드캐스트 병목
   → Redis Pub/Sub 또는 Kafka로 메시지 발행 분리 (비동기)
```
