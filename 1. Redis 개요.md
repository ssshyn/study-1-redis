
## 1. Redis

### 메모리 계층과 접근 속도
```
CPU Cache (1ns)
↓
RAM (100ns)
↓
SSD (0.1ms)
↓
HDD (10ms)
```


### Redis : RAM에 데이터 저장.
RDB에 비해 1,000배 이상 빠르긴 하나, 실제 RDB는 Buffer Pool에 자주 쓰는 데이터를 캐시하므로 항상 1,000배는 아니라고 한다.


### Single-Thread
: Redis는 싱글 스레드로 커맨드 처리를 한다.

##### 싱글 스레드지만 느리다 -> X
- 오히려 싱글 스레드이기 때문에 Race Condition 없이 Atomic 연산이 보장된다.

##### 싱글 스레드지만 빠른 이유
>  1. I/O Multiplexing ( epoll / kqueue ) :하나의 스레드가 수천 개의 소켓을 비동기로 감시함
>  2. 메모리 연산 : 디스크 IO가 없으니 스레드 컨텍스트 스위칭 비용이 병목이 안됨
>  3. 단순한 자료구조 : 복잡한 쿼리 파싱 없이 O(1) ~ O(N) 수준의 연산만 수행


### Redis의 데이터 영속성

| 방식                         | 설명          | 장단점              |
| -------------------------- | ----------- | ---------------- |
| **RDB** (Snapshot)         | 특정 시점 전체 덤프 | 복구 빠름, 사이 데이터 손실 |
| **AOF** (Append Only File) | 모든 쓰기 명령 로깅 | 데이터 안전, 파일 크기 큼  |
| **No Persistence**         | 캐시 전용       | 재시작 시 데이터 사라짐    |

## 2. 자료구조

### 1. String
- 가장 기본 타입 / Byte 배열로 저장되며, **최대 512MB**
- Redis의 String은 SDS(Simple Dynamic String)로 구현됨.
	- 길이를 O(1)로 조회
	- 바이너리 세이프 (null byte 포함 가능)
	- 미리 메모리를 더 할당해 append 시 재할당 최소화
- **주요 명령어**
```bash
  
  # 기본 set/get
  SET key value
  GET key
  
  # TTL 포함 저장
  SET key value EX 3600       # 3600초 후 만료
  SET key value PX 3600000    # 밀리초 단위
  
  # NX: Not Exist일 때만 저장 (분산 락 핵심)
  SET key value NX EX 30      # 30초
  
  # 숫자 증감 - Atomic 보장
  INCR counter         # 1 증가
  INCRBY counter 5     # 5 증가
  DECR counter         # 1 감소
  DECRBY counter 3     # 3 감소
  
  # 여러 키 한 번에
  MSET k1 v1 k2 v2
  MGET k1 k2
  
```

-  숫자를 String으로 저장, INCR / INCRBY 로 Atomic 하게 증가시킬 수 있다.


### 2. Hash

필드 - 값 쌍의 컬렉션. JAVA 기준 MAP<String, String> 과 유사하다.

Redis의 Hash는 데이터 크기에 따라 구조를 자동으로 전환한다.

### 내부 구현
#### #1. listpack
 : 아래 조건이 모두 만족할 때 사용된다.
   1. 필드 수 <= 128개 (hash-max-listpack-entries) 
   2. 모든 필드/값 길이 <= 64바이트 (hash-max-listpack-value)

- listpack은 연속된 메모리 블록에 데이터를 순서대로 나열함.
- 장점
	- 포인터가 없어서 메모리 오더헤드가 거의 없음 (포인터 1개 = 8byte)
	- 데이터가 연속 메모리에 있어 CPU 캐시 적중률이 높음
	- 필드 수가 적으면 선형 탐색(O(N))이어도 실제로 빠름
- 단점
	- 특정 필드를 찾으려면 앞에서부터 순서대로 스캔 > O(N)
	- 중간에 데이터 크기가 바뀌면 메모리 재할당 + 전체 복사 발생
	- 필드가 많아질수록 탐색 비용 선형 증가

#### #2. hashtable
: listpack 조건이 하나라도 초과되면 listpack -> hashtable 로 단방향 전환된다

```
버킷 배열
┌───┐    ┌──────────────────┐
│ 0 │───▶│ "name":"Alice"   │──▶ NULL
├───┤    └──────────────────┘
│ 1 │──▶ NULL
├───┤    ┌──────────────────┐    ┌────────────────┐
│ 2 │───▶│ "age":"30"       │───▶│ "role":"BIDDER"│──▶ NULL
├───┤    └──────────────────┘    └────────────────┘
│ 3 │──▶ NULL
└───┘
```
: 필드명을 해시 함수로 버킷 인덱스 계산, 충돌은 체이닝으로 해결

- 장점
	- 평균 O(1) 탐색/삽입/삭제
	- 필드 수가 아무리 많아도 성능 일정
- 단점
	- 버킷 배열 + 포인터로 인한 메모리 오버헤드 큼
	- 해시 충돌 시 성능 저하
	- CPU 캐시 비친화적 (포인터 따라가면서 메모리 여기저기에 접근함)

#### Rehashing - 확장 시 성능 보호

hastable의 load factor(로드 팩터 = 사용 중인 버킷 / 전체 버킷)가 일정 임계값을 넘으면 버킷 배열을 2배로 확장하고 전체 데이터를 재배치 함

Redis는 이 과정을 Incremental Rehashing 으로 처리함
```
ht[0] (기존): 크기 4
ht[1] (신규): 크기 8  ← 새 버킷 할당

모든 쓰기/읽기는 ht[0]와 ht[1] 모두 확인
커맨드 실행마다 ht[0]에서 ht[1]으로 N개씩 이전
이전 완료 후 ht[0] 해제
```
한 번에 전부 복사하면 Redis가 블로킹되므로, 조금씩 나눠서 이전하여 지연 스파이커를 방지

** 전환 시점 직접 확인하기 **
```bash
# Hash 내부 구조 확인
OBJECT ENCODING auction:room:1
# "listpack" 또는 "hashtable"

# 임계값 설정 확인
CONFIG GET hash-max-listpack-entries   # 기본 128
CONFIG GET hash-max-listpack-value     # 기본 64 (바이트)

# 임계값 변경 (운영 중 실시간 적용 가능)
CONFIG SET hash-max-listpack-entries 64
```


### 3. List

Double Linked List (실제로는 양방향 링크드 리스트가 아닌 listpack/quicklist).
순서 있는 컬렉션, 큐/스택 구현에 적합

#### 내부 구현
- 요소 수가 적으면 > listpack (연속 메모리, 캐시 친화적)
- 많으면 > quicklist (listpack의 연결 리스트, 각 노드가 listpack)
#### 주요 명령어
```bash
# 왼쪽(앞)/오른쪽(뒤) 추가
LPUSH queue task3 task2 task1   # 결과: [task1, task2, task3]
RPUSH queue task4               # 결과: [task1, task2, task3, task4]

# 꺼내기
LPOP queue    # task1 반환, 제거
RPOP queue    # task4 반환, 제거

# 블로킹 꺼내기 (메시지 큐에 유용함)
BLPOP queue 10   # 10초 동안 기다렸다가 데이터 오면 꺼냄

# 범위 조회 (제거 안함)
LRANGE queue 0 -1    # 전체 조회
LRANGE queue 0 4     # 인덱스 0 ~ 4

# 길이
LLEN queue

# 인덱스로 조회
LINDEX queue 0    # 첫번째거 조회

# 크기 제한 (오래된 것 자동 삭제)
LTRIM queue 0 99  # 최신 거 100개만 유지함

```

### 4. SET
중복 없는 무순서 컬렉션, Java의 HashSet과 유사함.
교집합, 합집합, 차집합 연산 지원

#### 내부 구현
- 요소 수가 적으면 > listpack
- 요소 수가 많으면 > hashtable

#### 주요 명령어
```bash
# 추가 / 제거
SADD roomUsers:1 "user_42" "user_15" "user_7"
SREM roomUsers:1 "user_7"

# 포함 여부
SISMEMBER roomUsers:1 "user_42"     # 1 또는 0
SMISMEMBER roomUsers:1 "user_42" "user_99"   # [1, 0]

# 전체 조회
SMEMBERS roomUsers:1

# 집합 연산
SUNION roomUsers:1 roomUsers:2  # 합집합
SINTER roomUsers:1 roomUsers:2  # 교집합
SDIFF roomUsers:1 roomUsers:2   # 차집합

# 크기
SCARD roomUsers:1

# 무작위 추출
SRANDMEMBER roomUsers:1 3  # 랜덤 3개 (제거 안 함)
SPOP roomUsers:1   # 랜덤 1개 꺼내고 제거
```


### 5. Sorted Set (ZSet)

각 멤버에 score(double)가 붙은 정렬된 집합. (SET + 정렬)
내부적으로 skiplist + hashtable로 구현되어 있음.

#### 내부 구현 - Skiplist
```
Level 3: head ---------> 25 -----------> tail
Level 2: head --> 10 --> 25 --> 50 ----> tail
Level 1: head --> 10 --> 25 --> 30 --> 50 --> tail
```
skiplist는 여러 레벨의 포인터를 두어 평균 O(log N)으로 삽입/삭제/검색이 가능하다.
일반 linked list의 O(N)보다 훨씬 빠르다.

#### 주요 명령어
```bash
# 추가 (score: double, member: string)
ZADD leaderboard 3500 "user_42"
ZADD leaderboard 2800 "user_15"
ZADD leaderboard 4100 "user_7"

# 옵션
ZADD leaderboard NX 5000 "user_99"   # 없을 때만
ZADD leaderboard XX 5500 "user_99"   # 있을 때만 업데이트
ZADD leaderboard GT 6000 "user_99"   # 기존보다 클 때만

# 범위 조회 (오름차순)
ZRANGE leaderboard 0 -1            # 전체 (낮은 점수 먼저)
ZRANGE leaderboard 0 -1 WITHSCORES # 점수 포함
ZRANGE leaderboard 0 -1 REV        # 내림차순 (높은 점수 먼저)

# 점수 범위로 조회
ZRANGEBYSCORE leaderboard 3000 5000 WITHSCORES

# 순위 조회 (0-based)
ZRANK leaderboard "user_42"      # 낮은 점수 기준 순위
ZREVRANK leaderboard "user_42"   # 높은 점수 기준 순위

# 점수 조회
ZSCORE leaderboard "user_42"

# 점수 증감 (Atomic)
ZINCRBYU leaderboard 500 "user_42"

# 삭제
ZREM leaderboard "user_15"
ZREMRANGEBYSCORE leaderboard 0 1000  # 점수 범위 삭
```

## 3. RedisTemplate 동작 원리

### 1. Lettuce vs Jedis
Spring Boot 2.x부터 기본 Redis 클라이언트는 Lettuce 다.

| 항목          | Lettuce                         | Jedis            |
| ----------- | ------------------------------- | ---------------- |
| 스레드 모델      | **비동기 Non-blocking** (Netty 기반) | 동기 Blocking      |
| 커넥션 공유      | 하나의 커넥션을 여러 스레드가 공유             | 스레드마다 커넥션 필요     |
| 커넥션 풀       | 기본적으로 불필요 (공유)                  | 필수 (`JedisPool`) |
| Reactive 지원 | `ReactiveRedisTemplate` 가능      | 불가               |
| 클러스터        | Automatic failover 지원           | 수동 처리 필요         |

> Lettuce가 커넥션을 공유하기 떄문에 커넥션 풀이 없어도 고성능이다.
> 다만, SUBSCRIBE 등 블로킹 커맨드를 쓸 때는 별도 커넥션이 필요하다. (StatefulRedisPubSubConnection)

### RedisTemplate 클래스 계층

```
RedisAccessor
  └── RedisTemplate<K, V>
        ├── StringRedisTemplate    (K, V 모두 String 고정)
        └── (커스텀 제네릭 타입 가능)
```

`RedisTemplate`은 Spring의 `JdbcTemplate`과 같은 역할:
- 커넥션 생명주기 관리
- 직렬화/역직렬화 처리
- 예외 변환 (`DataAccessException` 계층으로 통일)
- `execute(RedisCallback<T> action)` 로 네이티브 커넥션 접근 가능

### 내부 동작 흐름

```
코드: redisTemplate.opsForValue().set("key", value)
      ↓
1. RedisSerializer로 key/value 직렬화
      ↓
2. LettuceConnectionFactory에서 커넥션 획득
      ↓
3. Lettuce → Netty → TCP → Redis 서버
      ↓
4. Redis 서버: "SET key serializedValue"
      ↓
5. 응답 역직렬화 후 반환
```

## 4. 직렬화 전략

#### 4-1. 기본값의 문제점
- RedisTemplate<Object, Object>를 아무 설정 없이 쓰면 기본 직렬화는 JdkSerializationRedisSerializer 다.
```
Redis CLI에서 보면:
"\xac\xed\x00\x05t\x00\x05hello"
```

- 문제점
1. 가독성 없음 - Redis CLI로 디버깅 불가
2. Java 의존성 - Python, Go 등 다른 언어에서 읽을 수 없음
3. 크기가 큼 - 메타 데이터가 붙어서 메모리 낭비
4. 보안 취약 - Java 역직렬화 취약점 (CVE 계열)

#### 4-2. 직렬화 션 비교
| Serializer                           | 저장 형태            | 장점             | 단점                        |
| ------------------------------------ | ---------------- | -------------- | ------------------------- |
| `JdkSerializationRedisSerializer`    | 바이너리             | 기본값, 별도 설정 불필요 | 가독성 없음, 보안 취약             |
| `StringRedisSerializer`              | 문자열 그대로          | 가독성 최고, 가벼움    | String만 가능                |
| `GenericJackson2JsonRedisSerializer` | JSON (클래스 정보 포함) | 다양한 타입         | `@class` 필드 포함, 클래스 경로 의존 |
| `Jackson2JsonRedisSerializer<T>`     | JSON (클래스 정보 없음) | 깔끔한 JSON       | 타입 고정, 제네릭 처리 주의          |
| `GenericToStringSerializer`          | toString() 결과    | 숫자 타입에 유용      | 커스텀 파싱 필요                 |

#### 4-3. GenericJackson2JsonRedisSerializer의 @class 문제
```json
{
	"@class": "com.example.auction.domain.Bid",
	"amount": 150000,
	"userId": 42
}
```
클래스를 리팩토링해서 패키지 경로가 바뀌면 역직렬화 실패한다.
이를 해결하려면 `JsonTypeInfo` 커스텀 설정이나 별도 `RedisJsonSerializer`가 필요하다.

## 5. RedisConfig 분석

### RedisConfig.java
```java
@Configuration
public class RedisConfig {
	
	@Value("${spring.data.redis.host})
	private String host;
	
	@Value("${spring.data.redis.port})
	private int port;
	
	// 1. 커넥션 팩토리 - Lettuce 기반
	@Bean
	public RedisConnectionFactory redisConnectionFactory() {
		return new LettuceConnectionFactory(host, port);
	}
	
	// 2. RedisTemplate - 범용 (Object 직렬화)
	@Bean
	public RedisTemplate<String, Object> redisTemplate(
					RedisConnectionFactory connectionFactory) {
					
		RedisTemplate<String, Object> template = new RedisTemplate<>();
		template.setConnectionFactory(connectionFactory);
		
		// key는 String으로 저장 (가독성)
		template.setKeySerializer(new StringRedisSerializer());
		template.setHashKeySerializer(new StringRedisSerializer());
		
		// value는 JSON으로 저장
		Jackson2JsonRedisSerializer<Object> jsonSerializer =
			new Jackson2JsonRedisSErializer<>(Object.class);
		template.setValueSerializer(jsonSerializer);
		template.setHashValueSerializer(jsonSerializer);
		
		template.afterPropertiesSet();
		return template;
	}
	
	// 3. StringRedisTemplate - String 전용 
	@Bean
	public StringRedisTemplate stringRedisTemplate(
				RedisConnectionFactory connectionFactory) {
		return new STringRedisTemplate(connectionFactory);
	}
}
```

### RedisJsonSerializer.java 
- 프로젝트에 별도로 `RedisJsonSerializer`가 있는 경우, `ObjectMapper`를 커스텀해서 `@class` 없이도 타입을 안전하게 직렬화/역직렬화 하려는 목적

```java
public class RedisJsonSerializer<T> implements RedisSerializer<T> {
	
	private final ObjectMapper objectMapper;
	private final Class<T> type;
	
	public RedisJsonSerializer(Class<T> type) {
		this.type = type;
		this.objectMapper = new ObjectMapper()
			.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false)
			.registerModule(new JavaTimeModule()); // LocalDateTime 지원
	}
	
	@Override
	public byte[] serialize(T value) throws SerializationException {
		if (value == null) return new byte[0];
		try {
			return objectMapper.writeValueAsBytes(value);
		} catch (JsonProcessingException e) {
			throw new SerializationException("직렬화 실패", e);
		}
	}
	
	@Override
	public T deserialize(byte[] bytes) throws SerializationException {
		if (bytes == null || bytes.length == 0) return null;
		try {
			return objectMapper.readValue(bytes, type);
		} catch (IOException e) {
			throw new SerializationException("역직렬화 실패", e);
		}
	}
}
```

## 6. Operations API
`RedisTemplate`은 자료구조별로 `Operations` 객체를 제공한다.

### ValueOperations - String 타입
```java
@Service
@RequiredArgsConstructor
public class AuctionCacheService {
	
	private final RedisTemplate<String, Object> redisTemplate;
	private final StringRedisTemplate stringRedisTemplate;
	
	// === ValueOperations ===
	
	// 단순 저장/조회
	public void saveHighestBid(Long roomId, Long amount) {
		String key = "auction:room" + roomId + ":highestBid";
		// StringRedisTemplate이면 valueOps 바로 사용
		stringRedisTemplate.opsForValue().set(key, String.valueOf(amount));
	}
	
	// TTL 포함 저장
	public void saveWithTTL(String key, Object value, long seconds) {
		redisTemplate.opsForValue().set(key, value, Duration.ofSeconds(seconds));
	}
	
	// 원자적 증가
	public Long incrementBidCount(Long roomId) {
		String key = "auction:room:" + roomId + ":bidCount";
		return redisTemplate.opsForValue().increment(key);
	}
	
	// NX (없을 때만 저장) - 분산 락때 사용
	public Boolean setIfAbsent(String key, String value, long timeoutSeconds) {
		return redisTemplate.opsForValue()
			.setIfAbsent(key, value, Duration.ofSeconds(timeoutSeconds));
	}
	
	// 여러 키 한 번에
	public void multiSet(Map<String, Object> keyValueMap) {
```

## 7. TTL과 Eviction Policy
### TTL 설정 방법
```java
// 저장 시 TTL 설정
redisTemplate.opsForValue().set(key, value, Duration.ofHours(1));

// 이미 저장된 키에 TTL 추가
redisTemplate.expire(key, Duration.ofMinutes(30));

// TTL 조회
Long ttl = redisTemplate.getExpire(key, TimeUnit.SECONDS);

// TTL 제거 (영구 보관)
redisTemplate.persist(key);
```

### Eviction Policy (메모리 부족 시 삭제 전략)

`redis.conf`의 `maxmemory-policy` 설정:

| 정책             | 설명               | 사용 케이스            |
| -------------- | ---------------- | ----------------- |
| `noeviction`   | 메모리 꽉 차면 오류 반환   | 데이터 손실 절대 안 되는 경우 |
| `allkeys-lru`  | LRU로 어떤 키든 삭제    | 일반 캐시             |
| `volatile-lru` | TTL 있는 키만 LRU 삭제 | 캐시 + 영구 키 혼용      |
| `allkeys-lfu`  | LFU로 어떤 키든 삭제    | 특정 키가 집중 조회될 때    |
| `volatile-ttl` | TTL이 짧은 것부터 삭제   | TTL 중심 관리         |

### 9-1. Key 설계 - 네임스페이스

### 9-3. `@Transactional`과 Redis는 별개

Spring `@Transactional`은 **JPA/JDBC 트랜잭션** 만 관리한다.
Redis 작업은 별도로 `MULTI/EXEC` (RedisTemplate transaction)을 써야 원자성이 보장됨

```java
// 이렇게 해도 Redis 롤백 안 됨
@Transactional
public void bid(Long roomId, Long amount) {
	redisTemplate.opsForValue().set("key", amount); // DB 롤백해도 redis는 유지됨
	jpaRepository.save(entity);
	throw new RuntimeException("롤백!"); // DB만 롤백, Redis는 그대로
}
```
Redis + DB의 원자성이 필요하면 `Saga 패턴`이나 별도 보상 로직이 필요함..

#### Saga 패턴 보상 로직
```java
public void bid(Long roomId, Long amount) {

    String key = highestKey(roomId);

    Long previousAmount =
        (Long) redisTemplate.opsForValue().get(key);

    redisTemplate.opsForValue()
        .set(key, amount);

    try {

        bidRepository.save(
            new Bid(roomId, amount)
        );

    } catch (Exception e) {

        // 보상
        redisTemplate.opsForValue()
            .set(key, previousAmount);

        throw e;
    }
}
```
> 이런 식으로 exception 시 보상 데이터 저장으로 처리함.
> (다만 실무에서는 각 단계별로 listener로 분류하여 처리하는 경우가 잦음)

### 9-5. Connection Pool 설정 (고트래픽 시)
Lettuce는 기본적으로 커넥션을 공유하지만, `SUBSCRIBE`, `BLPOP` 같은 블로킹 커맨드가 많으면 별도 풀 설정 필요함
```java
@Bean
public LettuceConnectionFactory redisConnectionFactory() {
	LettuceClientConfiguration config = LettucePoolingClientConfiguration.builder()
		.poolConfig(new GenericObjectPoolConfig<>() {{
			setMaxTotal(20);
			setMaxIdle(10);
			setMinIdle(2);
		}})
		.build();
	return new LettuceConnectionFactory(
		new RedisStandaloneConfiguration(host, port), config);
}
```
